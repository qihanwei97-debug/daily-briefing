### 🔥 头条一：Anthropic 公开承认 Claude Code "一个月变笨"——3 个工程失误，全员用量重置

Anthropic 4 月 24 日发布工程博客，**正式承认 Claude Code 在 3 月初至 4 月中旬出现性能下降确为内部工程失误，与外界"模型被 quantize"或"刻意降智"的猜测无关**（[Fortune](https://fortune.com/2026/04/24/anthropic-engineering-missteps-claude-code-performance-decline-user-backlash/) | [VentureBeat](https://venturebeat.com/technology/mystery-solved-anthropic-reveals-changes-to-claudes-harnesses-and-operating-instructions-likely-caused-degradation) | [The Register](https://www.theregister.com/2026/04/23/anthropic_says_it_has_fixed/)）。三个独立 bug 累加：

1. **3 月 4 日：默认推理强度下调** ——Claude Code 默认 `effort` 从 `high` 调至 `medium` 以降低延迟，"权衡选错了"
2. **3 月 26 日：缓存 bug** ——模型在会话中持续丢弃自己的推理历史，导致"显得健忘+错乱"且**异常消耗用户配额**
3. **4 月 16 日：冗余度 prompt** ——系统层加了"工具调用之间回应限 25 字"指令，"实测损害编程质量"——4 天后撤回

**全部三个问题在 4 月 20 日已修复。API 全程不受影响（仅 Claude Code/Claude.ai 受影响）**。Anthropic 已于 **4 月 23 日为所有 Pro/Max 订阅者重置用量限额**，弥补 bug 期间的 token 浪费。

**为什么重要：** ① **Anthropic 第一次为产品质量公开 postmortem**——此前业界普遍怀疑 Claude 被"暗中降智"以扩容，本次披露给出了具体技术失误细节，**短期内是品牌信任修复，长期是 AI 产品工程透明度的新基准**；② **对律所等高强度 Claude 用户而言**——3 月初至 4 月中旬若发现"Claude 写合同/审核摘要质量异常"，可基本确认与 bug 期重合，需要重做关键产出的 QA；③ Opus 4.7 + Code 重大重构 + 用量重置组合拳，**实际上是 Anthropic 在 GPT-5.5 + DeepSeek V4 同周发布前的"拉回士气"动作**——5 月 Code 用户体验若显著优于 4 月，Claude Code 仍是最强 IDE 助手候选。

- 📍 获取：Claude Code `claude --update` 升级至 v2.1.119+；订阅用户登录 [claude.ai](https://claude.ai) 查看用量已重置

### 🔥 头条二：DeepSeek V4 vs GPT-5.5 vs Claude Opus 4.7——三强独立基准首批数据出炉

继 4/23-24 OpenAI GPT-5.5 + DeepSeek V4 同周发布后，**首批独立第三方基准已公布**，三强格局清晰浮现（[VentureBeat](https://venturebeat.com/technology/deepseek-v4-arrives-with-near-state-of-the-art-intelligence-at-1-6th-the-cost-of-opus-4-7-gpt-5-5) | [CodeRabbit GPT-5.5 实测](https://www.coderabbit.ai/blog/gpt-5-5-benchmark-results) | [OpenAI 官方介绍页（403 未 fetch）](https://openai.com/index/introducing-gpt-5-5/) | [DeepSeek V4-Pro Hugging Face 模型卡（403）](https://huggingface.co/deepseek-ai/DeepSeek-V4-Pro)）：

| 基准（人话解释） | DeepSeek V4-Pro | GPT-5.5 | Claude Opus 4.7 |
|------|-----------------|---------|-----------------|
| **SWE-bench Verified**（真实软件工程任务通过率） | 80.6% | ~78.2% | ~80.4% |
| **SWE-bench Pro**（更难版本，业界顶级 ~70%） | 55.4% | 58.6% | **64.3%** |
| **OSWorld-Verified**（操作系统 Agent 任务，人类基线 72%） | 未披露 | **78.7%** | 78.0% |
| **Terminal-Bench 2.0**（终端命令任务） | 未披露 | **82.7%** SOTA | 未披露 |
| **GDPval**（44 种白领工作产出质量） | 未披露 | **84.9%** | 未披露 |
| **API 定价**（input / output per M） | $1.74 / $3.48 | $5 / $30（Pro $30/$180） | $15 / $75 |

**关键结论：**
- **编程能力（SWE-bench Verified）三家几乎打平**——但越到难任务（SWE-bench Pro）Opus 4.7 优势越大
- **Agent 化能力（OSWorld）GPT-5.5 微幅领先 Opus 4.7**——这是 OpenAI"$5/$30 翻倍定价"的核心说服点，但差距仅 0.7 个百分点

- **DeepSeek V4-Flash $0.14/$0.28**——直接吞掉中端价位段

**为什么重要（律所视角）：** ① **批量长文档处理（合同、判例、研报）成本断崖式下降**——百页合同尽调用 V4-Flash 约 $0.50 vs Opus 4.7 约 $30，量大场景差距 60 倍；② **敏感客户数据**仍应走 Opus/GPT 闭源路径（合规优势 + Anthropic ASL-4 安全协议）；③ **Agent/Computer Use 工作流**——尽调初筛、合规自动化首选 GPT-5.5（OSWorld 微幅领先）；④ **复杂法律推理**仍 Opus 4.7 首选（SWE-bench Pro 64.3% 是难任务能力代理指标）。

## ⑤ 本期技巧：律所 AI 文书的 citation cross-check 强制工作流

**场景：** 你接了一份并购协议争议的诉讼文书撰写任务，对方主张某项免责条款在 X 州判例法下不可执行，你需要在 48 小时内拿出一份援引 15-20 条相关判例的法律备忘录。诱惑很大：让 Claude 或 GPT-5.5 直接生成"包含判例引用"的初稿，再用半小时校对一下。**Sullivan & Cromwell 4 月 18 日的紧急申请就是这样翻车的——顶级 biglaw 都没逃过 AI 幻觉**。

**为什么必须做：** AI 模型生成法律文书时会**编造看似合理的判例编号、年份、引文段落**——它们字面上格式正确（看起来就像 `Smith v. Jones, 123 F.3d 456 (2d Cir. 2019)`），但**整个判例可能根本不存在于任何司法管辖区**。Nebraska 律师 Greg Lake 4 月 15 日已被无限期吊销执照（57/63 条引用有缺陷），Sullivan & Cromwell 紧随其后。Q1 全美罚款累计已破 $145K。法院的容忍度正快速归零——**已从"罚款"升级到"吊销执照+纪律审查"**。

**操作路径（强制三步）：**

1. **生成阶段降权 AI**：让 AI 生成法律论证骨架与措辞，**但禁止 AI 凭空生成判例引用**——在 prompt 中明确"不要列举具体案例，只描述需要引用的判例类型与该判例所支持的法律命题"。
2. **引用阶段人手或权威库 fetch**：用 **Westlaw、LexisNexis、北大法宝、威科先行** 等权威法律数据库**手动检索**符合 AI 描述命题的判例，**逐条核对案号、年份、法院、引文页码**。
3. **交付前一遍机器复核**：用 Claude Code 或 ChatGPT 写个简短脚本（或人工逐条），**把每一条引用回查到权威库的 URL，确认 URL 仍可解析**。Claude Code v2.1.119 的 `/ultrareview`（云端多 Agent 并行评审）虽为代码评审设计，但同样的"并行核查"思路可手动套用——把 20 条引用分给 4 个会话，每个会话核 5 条，结果汇总。

**注意事项：** ① **不要相信"AI 生成 + AI 自查"** ——同一模型自查容易遗漏自己幻觉出的"合理"内容；② **客户端隐私**——如客户数据涉密，务必走 Claude for Work / Azure OpenAI 等企业版，不要用 ChatGPT Free/Plus 个人账户处理；③ **保留可审计 trail**——所内可考虑要求每份 AI 辅助文书附一份"AI 使用声明 + citation 核查清单"作内部留档。
