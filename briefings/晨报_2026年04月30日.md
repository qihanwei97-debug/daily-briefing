### 🔥 头条二：Musk v. OpenAI 庭审 Day 2-3——Musk 自承"傻瓜"，"$38M 换 $800B 公司"+"bait-and-switch"金句出炉

Elon Musk 在 4 月 29 日 OpenAI 庭审第二日（cross-examination 进入交锋）继续作证，连发两条庭审金句：**"我是傻瓜（I was a fool）"——"我给了他们 3800 万美元基本免费的资金，他们用这笔钱建成了一家 8000 亿美元的营利公司"**；并称 2023 年得知微软投资 OpenAI 100 亿美元后曾给 Sam Altman 发短信："这他妈到底怎么回事（What the hell is going on）"，定性为"**bait-and-switch**（先吊钩后掉包）"（[NPR Day 2 完整报道](https://www.npr.org/2026/04/29/nx-s1-5803833/elon-musk-testimony-openai-altman-day-2) | [CNN Business](https://www.cnn.com/2026/04/29/tech/musk-openai-altman-lawsuit-testimony-day-two) | [NBC News（火爆交叉询问）](https://www.nbcnews.com/tech/tech-news/elon-musk-openai-sam-altman-trial-day-two-testimony-cross-examination-rcna342660) | [Bloomberg](https://www.bloomberg.com/news/articles/2026-04-29/musk-recounts-losing-trust-in-motives-of-openai-s-other-leaders) | [CNBC live updates Day 3](https://www.cnbc.com/2026/04/29/musk-altman-live-updates-day-3-open-ai-trial.html) | [DNyuz Day 2 复盘](https://dnyuz.com/2026/04/29/elon-musk-blasts-openai-bait-and-switch-during-heated-day-2-testimony/)）。**英文一手庭审录音/transcript 未公开发布；NPR/CNN 等已 fetch 测试 403，下文据交叉报道整合。**

**关键庭审增量（vs 上期 Day 1 报道）：**

- **Musk 资金披露：** 原承诺 $1B，实际仅出资 $38M——"因为我对团队失去了信心"；以此重新计算"信托义务转换"的对价基础
- **2023 年微软 100 亿投资节点：** Musk 庭上呈递与 Altman 的短信记录，"What the hell is going on" 短信被列为情绪与判断节点证据；OpenAI 律师反驳称 Musk 当时已离开董事会三年、无信息权
- **OpenAI 律师攻击线：** 集中攻击 Musk **个人动机**——"是否因 xAI 与 OpenAI 直接竞争而 selectively 起诉"；Musk 在交叉询问中"combative"（NBC 用词），多次直接顶撞律师
- **Sam Altman 出庭排期：** 仍在后续周次（Week 2-3 之间）；OpenAI 团队**抗辩主轴预计是"非营利→营利转换符合特拉华法 + 加州 AG 备案要求"**
- **Larry Page "speciesist" 段：** Day 1 已披露（参见上期晨报），Day 2 Musk 进一步称这条对话是他与 Page 决裂、转而创立 OpenAI 的直接触发
- **法官 Yvonne Gonzalez Rogers** Day 1 警告 gag order 后，Musk Day 2 庭外社交媒体克制（无新发酵）

**为什么重要（律所视角）：** ① **金额对价的法律意义远超数字本身**——"$38M 实际出资"如被陪审团采信为**"信托对价不充分"**，则 Musk 主张"撤销营利转换"的请求权基础将被削弱（你只投了 $38M 凭什么撤销 $800B 公司？）；反过来，若法院将 OpenAI 章程视为**信托声明（trust declaration）**而非合同，**出资金额无关**——这条法律解释会成为庭审分水岭；② **"bait-and-switch" 措辞的诉讼策略价值**——Musk 律师把这一短信定性升级为**慈善欺诈（charitable fraud）**而非简单合同违约，**对所有曾以"非营利使命"募款然后调整结构的 AI/科技/Web3 项目（含 Anthropic PBC、xAI、Mistral、各类 DAO 基金会）形成判例风险敞口**——汉坤代表早期 AI 公司或 PBC 客户的，必须把"使命表述变更的法律边界"列入合规审查清单；③ **OpenAI 抗辩"特拉华+加州 AG 备案"路径如成立**，则 NGO 转 for-profit 走 AG 路径将获判例背书——**对中国基金会/慈善信托想要"商业化变现"的客户具有反向参考**（中国法下并无类似路径，但客户会问）；④ **本周四（4/30）Musk 第三日交叉询问**——届时若 OpenAI 律师能展示 Musk 内部 xAI 投资人 deck 中对 OpenAI 的负面表述，将形成对 Musk 信用的直接打击。

## ⑤ 本期技巧：用 Claude Code permissions 锁住"PocketOS 9 秒删库"在律所端的复现风险

**场景：** 你是汉坤一位带年轻律师的合伙人，所内本季度开始让团队用 Claude Code 直接编辑客户合同 docx、批量改写 reply-to-RFC 草稿、跑 citation cross-check，甚至已经有人接通了 MCP 让 Claude Code 直接读写 iManage / NetDocuments 这类律所 DMS 系统。本周 PocketOS 创业公司发生的事故（Cursor 跑 Claude Opus 4.6 处理 staging 凭据问题，**agent 自寻一个未限定范围的 API token，9 秒内调一次 Railway API 删除整套生产数据库 + 备份**）让你脊背发凉——同样的剧情如果换成 Claude Code 持有 iManage admin token、agent 在某次"清理重复文件"时调用 batch delete 接口，**律所 30 年沉淀的客户档案可能在 1 个 prompt 后被删空**。这不是危言耸听——PocketOS 创始人 Jeremy Crane 在事故公开声明中明确承认：**agent 没有问任何确认问题**。

**为什么好用：** Claude Code 内置的 permissions 系统**评估顺序是 deny → ask → allow → defaultMode**——deny 规则永远最优先，且**项目级 `.claude/settings.json` 可被 git 共享、强制全所应用**。正确配置后，**任何破坏性操作（rm、drop、delete、批量覆盖、向云控制台调高危 API）都会触发 ask 弹窗**，agent 无法绕过。这不是把 AI 关进笼子，而是把"高 blast-radius 工具"和"日常工具"用配置层强制隔开——同事仍能放手让 AI 写、改、查，但任何"删除/破坏"动作都必须经过**人手按一次 Enter**。律所核心价值在于客户文档与历史档案的不可丢失，permissions 系统是把 AI 价值与律所风控两端拉齐的最直接工具。

**操作路径（4 步，新建律所专用 settings 模板）：**

1. **建项目级 `.claude/settings.json`** 在团队共享的项目目录（如 `~/projects/汉坤合同主力库/.claude/settings.json`），按下面骨架写入——`deny` 数组里塞进所有破坏性 Bash 命令前缀和危险 MCP 工具，`ask` 数组覆盖中风险动作：
   ```json
   {
     "permissions": {
       "deny": [
         "Bash(rm -rf*)", "Bash(rm -fr*)",
         "Bash(* drop database*)", "Bash(* truncate*)",
         "Bash(git push --force*)", "Bash(git reset --hard*)",
         "mcp__imanage__delete_*", "mcp__netdocuments__bulk_delete*"
       ],
       "ask": [
         "Bash(rm *)", "Bash(mv *)", "Edit", "Write",
         "mcp__imanage__update_*", "mcp__netdocuments__update_*"
       ],
       "allow": ["Bash(ls*)", "Bash(cat*)", "Bash(grep*)", "Read", "Glob", "Grep"]
     }
   }
   ```
2. **MCP token 范围最小化（与上述 deny 配套）**——这是 PocketOS 事故的根本原因。所内连接 iManage / NetDocuments / 内部数据库的所有 MCP server，**每个 token 只授予完成单一职责所需的最小权限**——读凭据不能含写权限，写凭据不能含删除权限，删除权限独立 token 且只在人工执行的 ad-hoc 脚本中临时使用，不交给 AI agent
3. **commit `.claude/settings.json` 到所内 git，配合 `policySettings` 强制下发**——按 [Anthropic Claude Code Permissions 官方文档](https://code.claude.com/docs/en/permissions) policy 路径，IT 团队可在 `~/.claude-policy/managed-settings.json` 强制声明"任何项目下都禁用某类高危 Bash"，**个人 settings 无法覆写**——这是给"自己开了 `--dangerously-skip-permissions` 的同事"兜底
4. **每月跑一次"红队演练"**——故意让 Claude Code 处理含触发性 prompt 的测试任务（如"清理 ~/projects/test 下重复文件"），观察 deny 规则是否拦下；若拦不住立即升级。Anthropic 在 GitHub 公开过一段 v1.0.93 时期 deny 规则被绕过的历史 bug（[issue #6699](https://github.com/anthropics/claude-code/issues/6699)），当前 v2.1.123 已修复，但**永远要测，不要默认信任**

**注意事项：** 第一，**permissions 不是数据安全唯一一道闸**——必须同时做 (a) 客户档案的离线增量备份（每日，不与生产同 token）、(b) DMS 系统启用"删除前 24 小时软删除窗口"、(c) 关键操作的 IP 白名单。permissions 拦的是 agent 的**误用**，离线备份兜的是 agent 之外的**任何人为/系统失误**。第二，**`ask` 弹窗在自动化批处理中的体验问题**——若你让 agent 跑批量改写，`ask` 会被频繁触发干扰流程；正确做法是在批处理前**手动审视一次任务范围**、改用 `allow` 规则的细粒度模式（如 `Bash(rm /tmp/*.tmp)` 单个具体路径放行）；千万不要为了省事把 `ask` 改成 `allow`。第三，**配合本期 ⑤ 实施时仍需人工监督**——permissions 把破坏性动作降至"必须人按一次 Enter"，但律师本人按 Enter 前要看清楚 prompt 全文，agent 写 prompt 的措辞可能模糊（如"清理 staging 数据"实际指向生产）——**养成"看 path/path/path 三遍再 Enter"的习惯**。
