### 🔥 头条一：微软-OpenAI 解绑——AI 行业最大双边合约改写，"独家排他时代"终结

微软与 OpenAI 4 月 27 日联合宣布**修订自 2019 年以来的全部合作核心条款**，把这段曾被视为"AI 行业最重要联盟"的关系切换到"非独家+多云+营收上限"模式（[CNBC](https://www.cnbc.com/2026/04/27/openai-microsoft-partnership-revenue-cap.html) | [Bloomberg](https://www.bloomberg.com/news/articles/2026-04-27/microsoft-to-stop-sharing-revenue-with-main-ai-partner-openai) | [TechCrunch](https://techcrunch.com/2026/04/27/openai-ends-microsoft-legal-peril-over:-its-50b-amazon-deal/) | [Yahoo Finance](https://finance.yahoo.com/sectors/technology/article/microsoft-openai-rewrite-partnership-to-eliminate-exclusive-model-access-change-revenue-sharing-140522820.html) | [Interesting Engineering](https://interestingengineering.com/ai-robotics/microsoft-openai-partnership-restructuring)）。一手信源 [Microsoft 官方博客](https://blogs.microsoft.com/blog/2026/04/27/the-next-phase-of-the-microsoft-openai-partnership/)、[OpenAI 官方介绍页](https://openai.com/index/next-phase-of-microsoft-partnership/) 今日 fetch 均返回 403，下文内容来自三方报道交叉验证。

**核心条款变化：**

| 维度 | 原条款（2019-2026.4） | 新条款（2026.4.27 起） |
|------|-----------------------|------------------------|
| **微软对 OpenAI IP 授权** | 独家、排他 | **非独家**，期限延至 **2032 年** |
| **OpenAI 云服务部署** | Azure 独家承载 | Azure 仍优先，但**全产品线可上任意云**（AWS/Google Cloud） |
| **微软向 OpenAI 营收分成** | 持续支付 | **彻底停止** |
| **OpenAI 向微软营收分成** | 20% 比例长期支付 | 比例不变但**封顶 cap**，2030 年到期 |
| **AGI 触发条款** | 微软须就 OpenAI 自宣 AGI 做出商业反应 | **删除**——AGI 实现与否由独立专家组判定，不再绑定微软 |

**为什么重要：** ① **Azure 排他终结，OpenAI 与 AWS 的 500 亿美元算力协议从"潜在违约"转为合规**——OpenAI 4 月与 Amazon 签署的 $50B Azure-外算力订单终于落地（[TechCrunch](https://techcrunch.com/2026/04/27/openai-ends-microsoft-legal-peril-over-its-50b-amazon-deal/)），这条结构性变化是触发本次重写的直接动因；② **微软实质获得"AI 多供应商权利"**——可继续推 Anthropic Claude、Mistral、自研 MAI 系列，不再被独家锁定；③ **20% 营收分成封顶**对 OpenAI 是巨大现金流松绑——按当前 ARR ~$130B 估算，封顶后每年节省数十亿美元；④ **AGI 条款删除**移除了微软最大法律风险——原条款下 OpenAI 一旦"自宣 AGI"可能触发微软义务对赌，此次改由独立专家组判定彻底拆雷；⑤ **法律视角**：本次修订是教科书级的"超大型科技合约重写"案例——值得 M&A、TMT 业务部门以正式备忘录形式归档复盘，跨境 IP 授权 + 云服务非独家化 + 营收分成上限三件事的写法模板都是新的。

### 🔥 头条二：中国发改委正式叫停 Meta 20 亿美元收购 Manus——AI 领域首例外资安全审查全面禁令

国家发展改革委下属"**外商投资安全审查工作机制办公室**" 4 月 27 日就 **Meta 收购 Manus 项目作出安全审查决定**——**禁止外资完成本次收购，要求双方撤销已签署的交易**（[新浪财经](https://finance.sina.com.cn/roll/2026-04-27/doc-inhvyefa7758572.shtml) | [观察者网](https://www.guancha.cn/economy/2026_04_27_815117.shtml) | [CNBC](https://www.cnbc.com/2026/04/27/meta-manus-china-blocks-acquisition-ai-startup.html) | [Bloomberg](https://www.bloomberg.com/news/articles/2026-04-27/china-blocks-meta-s-2-billion-acquisition-of-ai-startup-manus) | [CNN](https://www.cnn.com/2026/04/27/tech/china-blocks-meta-manus-intl-hnk) | [TechCrunch](https://techcrunch.com/2026/04/27/china-vetoes-metas-2b-manus-deal-after-months-long-probe/)）。一手政府公告 [发改委政务公开页](https://zfxxgk.ndrc.gov.cn/web/iteminfo.jsp?id=20623) 今日 fetch 返回 403，下文据交叉报道还原。

**事件全貌：**

- **2025 年 12 月 30 日：** Manus（新加坡注册、华人创始团队、北京研发主力）宣布被 Meta **以约 20 亿美元收购**——Meta 史上第三大并购案，创始人肖弘将出任 Meta 副总裁
- **2026 年 1 月 8 日：** 商务部启动评估调查，重点核查"出口管制+技术进出口+外商投资"三方面合规
- **2026 年 4 月 27 日：** 发改委下属安全审查办公室作出**最终禁令**，要求撤销交易
- **行业揭示的三大触发因素：** ① **Manus 核心技术被认定为出口受限敏感类目**——未走中国出口审批程序；② **训练数据高度依赖境内数据**——存在数据出境合规风险；③ **外资收购环节本身**未满足外商投资法相关合规要求

**为什么重要（律所视角）：** ① **AI 领域首次援引"外商投资安全审查"全面禁令**——此前同类案件多为附条件批准（如剥离子业务、技术隔离），本案是教科书级"全面禁止+撤销"先例，对所有涉中国 AI 资产的跨境 M&A 客户敲响警钟；② **撤销路径极其复杂**——按 Odaily [独立解读](https://www.odaily.news/zh-CN/post/5210509)，**20 亿美元价款需退回、数据需隔离与删除、已 onboard 至 Meta 的 Manus 团队需脱离**，技术上"AI 收购怎么撤销"几乎无先例；③ **触及涉外业务两条红线**：(a) **AI 模型本身可能被列为"两用物项"出口管制对象**——后续中国 AI 公司被海外收购需提前申请出口许可，律所要为客户预警这条新合规路径；(b) **训练数据出境**——《数据出境安全评估办法》《促进和规范数据跨境流动规定》在 AI 训练场景下首次成为否决依据；④ **预计未来 6-12 个月，类似案件会显著增加**——任何"中国创始团队 + 海外注册主体 + 国内研发"结构（即典型 VIE/红筹+AI 公司）被外资收购，**都需重做"国家安全 + 数据出境 + 出口管制"三重合规评估**

- 📍 获取：[发改委政务公开页（403）](https://zfxxgk.ndrc.gov.cn/web/iteminfo.jsp?id=20623) | [新浪财经全文](https://finance.sina.com.cn/roll/2026-04-27/doc-inhvyefa7758572.shtml)
- 💡 上手：**汉坤跨境 M&A、出海合规、数据合规三个组今日起把本案纳入培训案例库**——尤其涉及"AI/数据中心/算力"标的的 outbound M&A 客户尽调清单需新增"安全审查+出口管制"专项；Meta 的法律团队可能寻求外部中国律所协助"撤销执行"——这是难得的逆向 inbound 业务窗口

## ⑤ 本期技巧：Claude Code `/team-onboarding` 把"老律师的 AI 工作流"打包给新人

**场景：** 你是汉坤一位习惯用 Claude Code 写法律备忘录的合伙人，过去三个月已经把"把客户长合同丢进 CLAUDE.md → 用 `/init` 建项目骨架 → 用自定义 skill 跑 citation cross-check → 用 MCP 连接威科先行检索"这套流程跑得顺手，准确率和效率都比手工高一截。新晋律师 Lucy 4 月 28 日刚入职，她的 Macbook 装好了 Claude Code，但面对你贡献到 git 的 30 多个项目目录，她不知道哪一个是"主力"，也不知道你常用哪些斜杠命令、哪些 MCP server、哪些 prompt 模板。**过去通常的做法是你抽两小时一对一带教**——但 Anthropic 4 月 10 日上线的 `/team-onboarding` 命令把这个时间成本基本降到零。

**为什么好用：** Claude Code v2.1.101 起内置的 `/team-onboarding` 命令会**扫描你最近 30 天的 Claude Code 使用记录**——所有会话、命令、MCP server 调用、subagent 调用、自定义 skill 触发，自动生成一份**可读的 Markdown 入职指南**，新人拿过来就能复现你的工作环境。它的根本价值在于"知识传承不再依赖手写文档"——你怎么用 Claude Code 写东西，它就把"怎么用"反向工程出来。律所内部最稀缺的就是"把高级合伙人的隐性工作流显性化"的能力，这条命令直接戳中。

**操作路径（4 步）：**

1. **在你的主力项目目录运行命令**——必须是你最近 30 天**真实用过 Claude Code 的项目**（首次用、空 30 天的项目跑出来是空指南）：
   ```
   cd ~/projects/汉坤合同主力库
   claude
   > /team-onboarding
   ```
2. **审阅生成的 onboarding 指南**——Claude Code 会输出一份 Markdown 文件，**包含你常用的斜杠命令、MCP 服务器列表、subagent 配置、典型 prompt 模板，以及推荐的 CLAUDE.md 内容**。**先自己读一遍删掉敏感信息**（具体客户名、未公开案号、内部 prompt 中的法律观点等）
3. **沉淀到 CLAUDE.md + 项目 settings**——按官方文档建议（[Claude Code Commands 参考](https://code.claude.com/docs/en/commands)），把生成指南中"稳定的规则"挑出来固化到项目根目录的 `CLAUDE.md`，把"团队共享的 MCP 配置"放进 `.claude/settings.json` 并通过 git 共享
4. **把指南交给新人**——Lucy 拿到这份指南后，按文中"项目结构-常用命令-MCP 配置"三段，30 分钟内就能跑出第一份合规的客户备忘录初稿

**注意事项：** ① **千万先脱敏**——指南会包含你过去 30 天的具体 prompt 措辞，里面可能含客户身份、案件细节、内部法律意见，公开前必须人工过一遍；② **`/team-onboarding` 不是 CLAUDE.md 替代品**——它是"我现在怎么用"的快照，不是"应该怎么用"的规范；正确做法是**用它发现稳定模式 → 提炼成 CLAUDE.md 规则**；③ **30 天窗口的局限**——如果你最近一个月没有真正深度用 Claude Code 处理本类项目，生成出的指南会偏单薄，建议先自己跑两周再传承。
