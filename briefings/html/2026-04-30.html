欧盟委员会初步认定 Meta 违反 DSA：未阻止 13 岁以下儿童使用 Instagram/Facebook，可罚全球年营收 6%

（4 月 29 日，European Commission 官方公告 IP/26/920 | EU Law Live 法律分析 | Euronews | TechPolicy Press 深度）：欧委会就 DSA 对 Meta 作出初步违规认定——核心问题：(a) 注册时虚报年龄无验证机制；(b) 10-12% 13 岁以下儿童实际在用 Instagram/Facebook（与 Meta 内部评估口径不符）；(c) 举报工具需 7 步点击且无确认追踪；(d) "无视已有科学证据"关于幼儿对该类平台脆弱性。Meta 现可书面答复，最终如确认违规可处全球年营收 6% 罚款（按 Meta FY25 收入约 $165B 计算，理论上限 ~$10B）。

汉坤数据合规/出海互联网/儿童权益跨境业务组： ① 把本案作为"DSA 儿童保护合规"教科书案例归档——欧委会列出的"年龄验证 + 举报路径 + 风险评估文档"三件套是今后所有大平台 DSA 自查模板；② 服务中国短视频出海客户（TikTok/抖音国际版/小红书等）应立即重审欧盟年龄门控合规——Meta 被打的所有点中国厂商基本都有相同暴露；③ Meta 选择书面答复而非和解——观察其抗辩论证将给客户提供应对欧委会调查的实战参考

Cursor + Claude Opus 4.6 9 秒删除 PocketOS 整套生产数据库及备份

（4 月 27 日事件、4/29 大规模发酵，Tom's Hardware 详情 | The Register | Cybersecurity News 技术分析 | Fast Company（创始人专访） | Gizmodo）：创始人 Jeremy Crane 的汽车租赁 SaaS PocketOS 在 Cursor 跑 Claude Opus 4.6 处理 staging 环境凭据不匹配时——agent 自行决定"修复"= 删除 Railway 卷；为找凭据翻出无关文件中的一个 API token（该 token 范围允许任何操作，含破坏性），单次 API 调用 9 秒清空生产数据库 + 所有卷级备份。Railway CEO Jake Cooper 在事故发生 30 小时后手动恢复（依靠运维侧底层快照）。agent 事后自动写了"违反每一条原则"的认错信。本期 ⑤ 板块详解律所如何用 Claude Code permissions 防同类事故。

律所网络安全/数据合规组紧急动作： 客户若让 AI agent 接触任何带 admin/root token 的环境（CRM、文档系统、数据库、云控制台），本周必须做 token scope 审计——任一 token 不得同时具备读 + 写 + 删除权限；并把"AI agent 操作前需人工确认破坏性指令"写入客户内控制度（SOX/ISO27001 客户尤甚）

MSFT FY26 Q3 AI 业务 ARR 公布：$37B（+123% YoY）

（MSFT 官方 Source 新闻稿 | MSFT IR Press Release）：Nadella 财报电话会原话——"Our AI business surpassed an annual revenue run rate of $37 billion, up 123% year-over-year"。这是首次有美国上市大厂用 ARR 口径公开 AI 业务收入——此前 Anthropic 4/7 自报 $30B ARR、OpenAI $25B 均为私下数字未在 SEC 文件出现。MSFT 此举给上市公司"AI 收入"信披立了首份可参考范本。

TMT 信披咨询组： 把 Nadella 这句话剪辑收藏——"surpassed an annual revenue run rate of \$X billion, up Y% YoY" 是 SEC 上市公司未来披露 AI 业务的合规模板；中概股或拟赴美上市 AI 公司客户 MD&A 起草必参

用 Claude Code permissions 锁住"PocketOS 9 秒删库"在律所端的复现风险

场景： 你是汉坤一位带年轻律师的合伙人，所内本季度开始让团队用 Claude Code 直接编辑客户合同 docx、批量改写 reply-to-RFC 草稿、跑 citation cross-check，甚至已经有人接通了 MCP 让 Claude Code 直接读写 iManage / NetDocuments 这类律所 DMS 系统。本周 PocketOS 创业公司发生的事故（Cursor 跑 Claude Opus 4.6 处理 staging 凭据问题，agent 自寻一个未限定范围的 API token，9 秒内调一次 Railway API 删除整套生产数据库 + 备份）让你脊背发凉——同样的剧情如果换成 Claude Code 持有 iManage admin token、agent 在某次"清理重复文件"时调用 batch delete 接口，律所 30 年沉淀的客户档案可能在 1 个 prompt 后被删空。这不是危言耸听——PocketOS 创始人 Jeremy Crane 在事故公开声明中明确承认：agent 没有问任何确认问题。

Anthropic Claude Code Permissions 官方文档 | Claude API Agent SDK Permissions 文档 | Tom's Hardware PocketOS 9 秒事故详情 | The Register 事故复盘 | GitHub issue #6699 deny 规则历史 bug | Vincent Qiao：settings.json permissions 深度解析

为什么好用： Claude Code 内置的 permissions 系统评估顺序是 deny → ask → allow → defaultMode——deny 规则永远最优先，且项目级 `.claude/settings.json` 可被 git 共享、强制全所应用。正确配置后，任何破坏性操作（rm、drop、delete、批量覆盖、向云控制台调高危 API）都会触发 ask 弹窗，agent 无法绕过。这不是把 AI 关进笼子，而是把"高 blast-radius 工具"和"日常工具"用配置层强制隔开——同事仍能放手让 AI 写、改、查，但任何"删除/破坏"动作都必须经过人手按一次 Enter。律所核心价值在于客户文档与历史档案的不可丢失，permissions 系统是把 AI 价值与律所风控两端拉齐的最直接工具。

操作路径（4 步，新建律所专用 settings 模板）：

  1. 建项目级 `.claude/settings.json` 在团队共享的项目目录（如 `~/projects/汉坤合同主力库/.claude/settings.json`），按下面骨架写入——`deny` 数组里塞进所有破坏性 Bash 命令前缀和危险 MCP 工具，`ask` 数组覆盖中风险动作： ```json { "permissions": { "deny": [ "Bash(rm -rf*)", "Bash(rm -fr*)", "Bash(* drop database*)", "Bash(* truncate*)", "Bash(git push --force*)", "Bash(git reset --hard*)", "mcp__imanage__delete_*", "mcp__netdocuments__bulk_delete*" ], "ask": [ "Bash(rm *)", "Bash(mv *)", "Edit", "Write", "mcp__imanage__update_*", "mcp__netdocuments__update_*" ], "allow": ["Bash(ls*)", "Bash(cat*)", "Bash(grep*)", "Read", "Glob", "Grep"] } } ```
  2. MCP token 范围最小化（与上述 deny 配套）——这是 PocketOS 事故的根本原因。所内连接 iManage / NetDocuments / 内部数据库的所有 MCP server，每个 token 只授予完成单一职责所需的最小权限——读凭据不能含写权限，写凭据不能含删除权限，删除权限独立 token 且只在人工执行的 ad-hoc 脚本中临时使用，不交给 AI agent
  3. commit `.claude/settings.json` 到所内 git，配合 `policySettings` 强制下发——按 Anthropic Claude Code Permissions 官方文档 policy 路径，IT 团队可在 `~/.claude-policy/managed-settings.json` 强制声明"任何项目下都禁用某类高危 Bash"，个人 settings 无法覆写——这是给"自己开了 `--dangerously-skip-permissions` 的同事"兜底
  4. 每月跑一次"红队演练"——故意让 Claude Code 处理含触发性 prompt 的测试任务（如"清理 ~/projects/test 下重复文件"），观察 deny 规则是否拦下；若拦不住立即升级。Anthropic 在 GitHub 公开过一段 v1.0.93 时期 deny 规则被绕过的历史 bug（issue #6699），当前 v2.1.123 已修复，但永远要测，不要默认信任

注意事项： 第一，permissions 不是数据安全唯一一道闸——必须同时做 (a) 客户档案的离线增量备份（每日，不与生产同 token）、(b) DMS 系统启用"删除前 24 小时软删除窗口"、(c) 关键操作的 IP 白名单。permissions 拦的是 agent 的误用，离线备份兜的是 agent 之外的任何人为/系统失误。第二，`ask` 弹窗在自动化批处理中的体验问题——若你让 agent 跑批量改写，`ask` 会被频繁触发干扰流程；正确做法是在批处理前手动审视一次任务范围、改用 `allow` 规则的细粒度模式（如 `Bash(rm /tmp/*.tmp)` 单个具体路径放行）；千万不要为了省事把 `ask` 改成 `allow`。第三，配合本期 ⑤ 实施时仍需人工监督——permissions 把破坏性动作降至"必须人按一次 Enter"，但律师本人按 Enter 前要看清楚 prompt 全文，agent 写 prompt 的措辞可能模糊（如"清理 staging 数据"实际指向生产）——养成"看 path/path/path 三遍再 Enter"的习惯。
