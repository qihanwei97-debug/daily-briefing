用 Claude Code 的 Monitor 工具 + `/loop` 自适应间隔，把"长时任务巡检"交给 AI

律所做大型尽调或多方合同协同时，常常会有一个跑得很久的后台流程在你看不见的地方推进——比如批量 OCR 几千份合同、跑一轮带 citation check 的全量法规比对、或者等远端 e-discovery 任务给出结果。过去要么开个终端守在那里看日志、要么定个闹钟回来刷新。Claude Code 4 月新出的 Monitor 工具 + `/loop` 自适应间隔这对组合，正是给这种"启动一个长任务，过段时间来看看，发现问题就改"的场景设计的。

它好用的核心是把"看日志-判断-决定下一步"这条链路从你身上挪到 Claude 身上：Monitor 把后台进程的 stdout 流成对话事件，Claude 看到关键信号才打扰你；`/loop` 则把这种巡检按你给的间隔（不指定时按任务节奏自适应）持续跑，省得你每隔半小时手动 `tail -f`。对律师来说，这不是多写几个 prompt 的边际优化，而是把"等结果"这种纯监工时间换成"看到异常被推送过来再介入"——晚上挂着一个长跑的合同审查作业，第二天早上要么是干净结果、要么是一份 Claude 整理好的"以下三个文件命中风险条款，建议人工复核"清单。

操作路径上：① 先把 Claude Code 升到当前 4 月版（`claude --update`，需要 Pro 或 Max 订阅）；② 启一个长时任务时（比如 `python batch_review.py > run.log 2>&1 &`），让 Claude 调用 Monitor 工具盯住 `run.log`；③ 如果是周期性巡检（每小时检查一次中间结果），直接 `/loop` 不带参数让它自适应间隔，或 `/loop 30m <子命令>` 显式指定；④ 在 CLAUDE.md 里写清楚"什么模式算异常需要打断我"——比如"出现 ERROR、出现 citation_mismatch、连续 30 分钟无进度"。

注意事项有两条：第一，Monitor 流出来的内容会进入对话上下文，跑很久的任务务必让 Claude 周期性总结+压缩，否则会撞到上下文上限；第二，严禁把含可识别客户信息的文件路径或日志直接喂给云端 Claude——监控前先做脱敏，或者让 Monitor 只抓 hash/计数等元信息。来源：Claude Code What's New（官方）。
