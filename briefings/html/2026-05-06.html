头条一：GPT-5.5 Instant 5/5 成 ChatGPT 默认模型——医疗/法律/金融幻觉率降 52.5%，并支持跨会话/文件/Gmail 引用

OpenAI 5 月 5 日（周二）正式推送 GPT-5.5 Instant，作为 ChatGPT 全部用户（Free/Plus/Pro/Business/Enterprise）的新默认模型，替代此前的 GPT-5.3 Instant，并以 `chat-latest` 名义在 API 同步开放TechCrunch | Axios | 9to5Mac | Decrypt | MacRumors | Tech Portal。OpenAI 官方介绍页 openai.com/index/introducing-gpt-5-5 今日 fetch 返回 403，未能拿到一手页内容——下文据 6 家英文媒体交叉报道还原。

核心数据（来源：OpenAI 官方对各家媒体披露）：

维度关键数字 / 改动
高风险 prompt 幻觉率下降 52.5%（vs GPT-5.3 Instant），OpenAI 明确点名"medicine、law、finance"为测试领域
挑战性对话准确率用户标记为"事实错误"的难题，错误声明再降 37.3%
风格变化回答"更紧凑、直击要点"；减少装饰性 emoji、不必要的格式化、追加问题
新功能（Plus/Pro 网页端）可用搜索工具回看历史会话、上传文件、Gmail 内容，给出更个性化的回答；移动端"近期推送"
API 名称`chat-latest`（新订阅默认）
GPT-5.3 Instant 退役计划付费用户保留 3 个月通过 model configuration settings 切换；之后下线
可用性全量推送中，含 Free 层；Codex 中 GPT-5.5 / 5.5 Pro 仍按已有节奏
为什么重要（律所视角）：
  • OpenAI 主动把"law"列为重点优化场景——这是 frontier model 厂商首次在 release notes 把法律领域与医疗、金融并列作为幻觉率下降的主诉求场景；与 4/30 Sullivan & Cromwell 单文件 40+ 处 AI 幻觉的紧急道歉信形成鲜明对比，头部所即使用旧版 GPT-5.3 Instant 也踩坑，今后 prompt 默认升 5.5 后起点更高，但 citation cross-check 工序绝不能省——本周内向所内同事广播一次：默认模型已切换、行为可能与上周略有差异
  • 跨会话/文件/Gmail 引用是律师工作流升级点——一个长期诉讼项目里几十次咨询历史（脱敏后）能被自动串联，避免"每次重新解释案件背景"，但也带来新数据面——Plus/Pro 个人账号若与所内合伙人共用须立刻分账；客户敏感信息进 Gmail 引用范围更需脱敏
  • 52.5% 这个数字应"打七折用"——OpenAI 自评数字，外部独立基准未跟进；律所内部应建立"5.5 时代"的 citation 抽样审计机制，把样本量做出来再调整使用规范
  • API `chat-latest` 在企业合同里的口径——律所代表 AI 客户起草合同时，"指定模型版本 vs 自动跟随 latest" 是新条款焦点（latest 推送频次加快 → SLA、责任分配需要重新约定），可以把这次 GPT-5.5 Instant 的"无 opt-out 默认替换"作为典型条款示例归档
  • 📍 获取：ChatGPT 网页/App（已默认）；API 端 `chat-latest`；OpenAI 官方博客 introducing-gpt-5-5 fetch 403，未见官方一手页公开访问
  • 💡 上手：律所 IT 与合规组： 本周内对全所发"默认模型已切换"内部公告，附 1 页"行为差异提示 + 跨会话引用功能脱敏须知"；诉讼/争议解决组： 拿一个真实案件的 5 段公开 prompt 同时跑 5.3 Instant 与 5.5 Instant（前者还有 3 个月窗口），手工标注幻觉变化作所内培训案例

  • Google Cloud +63% 的合规外溢效应——上期"Google-DoD 任意合法政府用途"协议在本季加速兑现，对所内涉外政府客户（含中国企业海外子公司、含被 DoD 列采购清单的供应商）意味着"避开 Google Cloud"的成本越来越高，须在 5 月内为客户出"AI 云供应商替代清单 v2"——含 Anthropic、Oracle、Nebius、CoreWeave 选项及合规权衡
  • AGI 触发条款重写已落地——上周 MSFT-OAI 解绑后，本季 Microsoft 财报首次不再披露与 OpenAI 的营收分成，CFO Amy Hood 措辞偏紧——TMT 组复盘"独家排他到非独家"的合同重写如何映射到当季报表是新的实务材料

本期技巧

用 mattpocock/skills 的 `/grill-with-docs` 模式给律所合同起草做"前置质询"

场景： 你是汉坤一位带年轻律师的合伙人，每周要为外资客户起草或审一批跨境合同——分销协议、SaaS 订阅条款、关联交易合同、ESOP 文件。过去三个月所内同事用 Claude Code/ChatGPT 起草初稿越来越熟练，但你发现一个反复出现的低级错误：AI 在没真正读懂"谁是甲方、谁付钱、何时交付、何种法域、何种争议解决路径"之前就开始写条款——结果产出貌似流畅，里头却把"客户的关联方"当成了"客户本人"，把"交付里程碑触发付款"写成了"按月固定付款"。Sullivan & Cromwell 4/30 那封"求别罚"信背后的核心问题之一就是这种"AI 没读懂语境就开始生成"——单文件 40 处虚假 citation 之前，是 40 处对案件背景的隐式假设错误。本周 GitHub 热门的 `mattpocock/skills` 项目里有一条名叫 `/grill-with-docs` 的 skill，正是给这个问题专门设计的——它的英文原意是"用 docs 拷问（AI 自己）"，强迫 AI 在动笔前先做一次完整的"领域模型重述+追问"。把这条 skill 改写成法律版本，对律师工作流是即插即用的升级。

为什么好用： mattpocock/skills（一周新增 48K+ stars，连续 6 天 GitHub Trending #2）的核心创新是单文件 SKILL.md + YAML frontmatter 的极简架构——每个 skill 都是一段 prompt 模板加一两个 helper script，不依赖任何 MCP server、不依赖任何外部依赖，clone 即用。`/grill-with-docs` 的本质是把"程序员让 AI 在写代码前先画领域模型 + 列假设 + 自己提十个反问"这套工程实践，抽象成一个可复用的 prompt 模板。映射到律所工作流：让 AI 在起草任何条款前先用结构化方式重述客户 brief——主体、付款流、交付里程碑、违约救济、法域、纠纷解决——并且主动列出"我目前还不确定的 5 个事实"+"如果某假设错了，会导致哪些条款偏离"。这一步只花 2-3 分钟，但能把后续幻觉与隐式假设错误降至少一半。它的根本价值不在"提示词写得多巧"，而在用一个固定流程，强制 AI 把推理链条暴露出来——你能在它写第一行条款前就发现它的误解，而不是在审稿时反复纠错。

操作路径（5 步，30 分钟内完成所内首发版）：

  1. 下载 mattpocock/skills 仓库：在所内的 git 集中目录运行 `git clone https://github.com/mattpocock/skills.git`，重点看 `skills/engineering/grill-with-docs/SKILL.md` 的 frontmatter+正文结构——这是改写模板
  2. 新建 `skills/legal/grill-contract-brief/SKILL.md`：保留 mattpocock 的 YAML frontmatter 结构（`name`, `description`），把正文改成法律语境——参考骨架："Before drafting any clause, restate (a) parties and their relationship; (b) payment flow and triggers; (c) delivery milestones; (d) governing law and jurisdiction; (e) dispute resolution mechanism. Then list (i) 5 facts you're uncertain about; (ii) for each uncertain fact, which clauses depend on it; (iii) which side bears risk if the assumption flips. Wait for human confirmation before drafting."——可以用中英双语写 SKILL.md，所内年轻律师中英文项目都能用
  3. 接入到 Claude Code：把 `skills/legal/` 路径加入项目 `.claude/settings.json` 的 `skillPaths` 配置（按 Claude Code Settings 官方文档），所内同事 clone 项目后自动可用 `/grill-contract-brief`
  4. 首次使用流程：律师拿到客户 brief 后，第一步永远是 `/grill-contract-brief`——不要直接 `/draft-clause`；让 AI 跑一遍 5 维度重述+5 项不确定+对应风险条款映射；人类律师对照 brief 修改 AI 的理解，确认无误后再让 AI 起草初稿
  5. 每月 retro：5 月底开第一次所内 retro 会议——抽 10 个用 `/grill-contract-brief` 起草的合同，对比"用之前 vs 用之后"的初稿幻觉率与 senior 律师返工次数，把数据贡献回所内 Skills 仓库的 `LESSONS.md`

注意事项： 第一，这条 skill 替代不了人类律师的 brief 解读责任——它的价值是把"AI 的错误理解暴露在起草之前"而不是"AI 自动正确理解"，每次律师本人必须读一遍 AI 重述并主动修正；第二，与上期推荐的 Karpathy CLAUDE.md 配合使用——CLAUDE.md 是项目级行为约束，`/grill-contract-brief` 是任务级工作流，两者不冲突；第三，首次推行务必由合伙人用真实合同跑两周再扩散到团队，因为 prompt 的措辞要根据所内合同类型微调——分销合同与 ESOP 的"5 项不确定"不在同一维度上；第四，任何包含客户身份/案件标的/未公开商业条款的真实 brief，进入 prompt 前必须脱敏——参考上期 ⑤ 板块 Karpathy CLAUDE.md 法律版的脱敏规则。
