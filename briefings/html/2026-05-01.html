智谱 GLM-5 推理工程突破：LayerSplit KV Cache 分层方案吞吐 +10-132%，修复方案被 SGLang 上游开源社区采纳

（4 月 30 日凌晨智谱技术博客《Scaling Pain：超大规模 Coding Agent 推理实践》，网易转载详情 | 搜狐转载（含 SGLang 采纳细节） | 21 经济网（智谱"全球大模型第一股"重大发布））：智谱（02513.HK）首次系统披露 GLM-5 系列在超大规模 Coding Agent 调用场景下的推理工程突破——核心创新 LayerSplit：针对 Context Parallel 策略中 KV Cache 冗余存储问题，每张 GPU 仅持有部分层 KV Cache，通过广播机制完成协同计算；在 Cache 命中率 90% 条件下，40K 至 120K 请求长度区间内系统吞吐量提升 10-132%（KV Cache：模型推理时缓存的注意力中间结果，长上下文场景下占显存大头；Context Parallel：将长 prompt 切片到多 GPU 并行处理）。同时披露两个底层 bug 定位修复已被 SGLang 上游开源社区接收合并（SGLang：UC Berkeley 发起的高性能 LLM 推理框架，与 vLLM 同级别），是中国 AI 厂商技术贡献首次进入 LLM 推理领域国际开源主流。

用 mattpocock/skills 给律所的"半成型 AI 工作流"加上工程级护栏——尤其是 `/grill-with-docs` 与 `/tdd`

场景： 你是汉坤一位带 5-8 名年轻律师的合伙人，过去两个月所内已经把 Claude Code 用于合同条款修改、法律备忘录草拟、citation 核查。问题来了——年轻律师给 AI 的 prompt 普遍写得太开放："帮我改一下这条 indemnification 条款"——AI 自然会往大改方向走，可能把客户原本认可的措辞也"顺手优化"掉。同时你审完一份 AI 改的备忘录后很难知道自己具体审了什么——AI 没有事先列验收标准，你和它之间没有"已验证 vs 未验证"的明确分界。本周登顶 GitHub Trending 的 mattpocock/skills（周新增 30,945 star，超越上期 Karpathy-skills 的 23K）正好提供了两条针对性 skill——`/grill-with-docs`（开发前对齐）和 `/tdd`（先列验收标准再循环验证）——这两条思路一字不改地搬到法律工作场景，能立刻消除上述两类痛点。

mattpocock/skills（GitHub 主仓库，fetch 通过） | GitHub Trending Weekly 实测数据（fetch 通过） | Anthropic Claude Code Skills 官方文档 | Anthropic Claude Code Settings 官方文档 | 上期 4/29 ⑤ Karpathy-skills 框架（互补使用） | 上期 4/30 ⑤ Permissions 防删库技巧（与 `/tdd` 自检清单打通）

为什么好用： Karpathy-skills 解决的是"AI 行为契约的总原则"（Simplicity / Surgical Changes / Clarity / Goal-Driven）；mattpocock/skills 走得更远——把这些原则拆解为可调用的 slash command，让 AI 在每次任务开始时强制走"先对齐、后验证"的流程。`/grill-with-docs` 的核心动作是：AI 不会立刻动手，而是反复用具体问题（参考相关 docs）"烤"提问者，直到对齐颗粒度足够细才进入执行；`/tdd` 的核心是：先把任务拆为"成功标准 1、2、3"清单，每完成一条立刻自检通过/不通过、不通过则回到上一步。两条结合起来正好对应律师工作的两个核心痛点——"客户和律师的需求颗粒度不一致"（前者）+"律师审核 AI 产出时缺乏验收标准"（后者）——而且它的安装就是一行 `npx` 命令、与所内已有 CLAUDE.md（Karpathy 框架）完全兼容、可独立 commit 到 git 共享，技术成本几乎为零。

操作路径（4 步，30 分钟内完成）：

  1. 安装 mattpocock/skills 到所内主力项目目录： 在 `~/projects/汉坤合同主力库` 或类似团队共享目录执行 `npx skills@latest add mattpocock/skills` ——会自动在 `.claude/skills/` 下生成全套 skill 文件并注册到 Claude Code；与上期 ⑤ 已部署的 Karpathy CLAUDE.md 不冲突
  2. 修改条款类任务统一走 `/grill-with-docs` 启动： 律师在 Claude Code 中开始任何"修改/改写"类任务前，先输入 `/grill-with-docs <任务一句话描述>`——AI 会反向追问 5-10 个细节（如"客户原文哪些措辞不能动？""赔偿上限是否有商业谈判余地？""管辖法适用何处？"），律师必须把答案给齐才进入执行——这一步等于把'客户认可的原文'白纸黑字钉在上下文里，AI 后续不会越界改动
  3. 审核类任务统一走 `/tdd` 验收： 当 AI 完成一份草稿后，律师输入 `/tdd 列出本备忘录应满足的 5 条验收标准并逐条自检`——AI 会先列清单（如"引用判例必须从 Westlaw 粘贴""赔偿条款必须保留客户原措辞""时区表述不得擅自标准化"），然后逐条标注通过/未通过，未通过的部分要求律师手动修复或确认放行——这一步等于把过去律师"凭感觉审"升级为"按清单审"
  4. 配合 `/diagnose` 处理"AI 出错怎么办"： 当律师发现 AI 产出有具体问题（如引文格式错误、条款编号跳号），输入 `/diagnose <具体问题>`——AI 会按结构化方法找根因（vs 直接打补丁式硬改），找到根因后再让 AI 给出"最小修改方案"——对法律文书的可追溯性极重要

注意事项： 第一，`/grill-with-docs` 在律师角度可能感觉"啰嗦"——AI 会问 5-10 个问题，每问一遍要回一遍，初次使用时律师容易嫌烦直接跳过；正确做法是前两周强制使用，之后保留"低风险任务可跳过、客户重要意见书必走"的内部规则——若所内年轻律师跳过该步骤导致 AI 越界改动，应在协作 retro 中复盘；第二，mattpocock 本人是 TypeScript / TDD 教育者，原 skills 部分内容（如 git guardrails、test migrations）对律师无用——保留无碍但不要因为"看不懂"就否定整套，重点用 `/grill-with-docs` `/tdd` `/diagnose` `/triage` 这 4 条；第三，与 4/30 ⑤ 的 permissions 防删库技巧打通——`/tdd` 自检清单中可固定加一条"本任务是否触及删除/批量覆盖类操作？若是则强制人工 Enter 确认"——把行为约束（mattpocock skills）与权限约束（Claude Code permissions）两层串起来，对律所核心档案的保护是最稳妥的；第四，4/30 临近 API 大变化窗口期——若所内同步升级到 Claude Code v2.1.126，安装 mattpocock/skills 后建议重启 Claude Code 让 `/skills` 命令重建索引
