### 🔥 头条二：SoftBank 拟分拆"Roze AI"赴美 IPO，估值目标 $100B——主业"机器人化建数据中心"，打包 ABB 机器人 + Ampere + DigitalBridge 三宗已收购资产

SoftBank Group 4 月 29-30 日据 Financial Times 首发报道、TechCrunch / CNBC 跟进证实：**孙正义正在主导一次旗下 AI 资产大整合**——新设实体名为 **Roze**，将申请赴美 IPO，**目标估值约 $1000 亿美元**，**最快 2026 年下半年挂牌**（[TechCrunch（FT 转引）](https://techcrunch.com/2026/04/29/softbank-is-creating-a-robotics-company-that-builds-data-centers-and-already-eyeing-a-100b-ipo/) | [CNBC](https://www.cnbc.com/2026/04/30/softbank-roze-ai-robotics-ipo-100-billion-ft-report.html) | [Yahoo Finance](https://finance.yahoo.com/markets/stocks/articles/softbank-plans-list-ai-robotics-003609305.html) | [Parameter 战略分析](https://parameter.io/softbank-group-bets-on-100b-roze-to-scale-robot-driven-data-centers/) | [TechFundingNews](https://techfundingnews.com/masayoshi-son-softbank-roze-ai-100bn-ipo/) | [Mezha（乌克兰科技媒体复盘）](https://mezha.net/eng/bukvy/916a162e_softbank_plans_roze/)）。**FT 原报道页与 SoftBank 官方 IR 均 fetch 403，未见 SoftBank 官方公告 URL fetch 通过**——下文据 TechCrunch / CNBC / Parameter 多家二手交叉整合；SoftBank 官方截至本期截稿暂未发布对应英文 PR。

**Roze 资产打包结构（据 FT/CNBC 披露）：**

| 已收购资产 | 收购对价 | 业务定位 |
|------------|---------|---------|
| **ABB 机器人事业部** | $5.4B（2026 年初公告） | 工业自动化机械臂、物流机器人 |
| **Ampere Computing**（Arm 服务器 CPU 设计公司） | $6.5B（2025 公告） | 数据中心 Arm 架构服务器芯片 |
| **DigitalBridge** | $3B（基础设施基金子合伙人权益） | 数据中心 / 通信塔 / 边缘基础设施 REIT |
| **新增"重型/焊接/装配自主机器人"团队** | 内部组建 | 数据中心建造现场施工自动化 |

**业务模式：** Roze 的核心定位**不是"卖机器人"**，而是用自有机器人队伍**自建数据中心给 hyperscaler 客户租赁**——以"机器人替代建筑工人 → 缩短数据中心建造周期 → 降低 AWS/Azure/GCP 客户算力价格"为商业逻辑。**估值锚点是与 5 家 hyperscaler $700B/年 Capex 形成"上游施工卡位"的供应商溢价**。

**为什么重要（律所视角）：** ① **AI 基础设施 IPO 范式正在分化**——传统 AI 公司 IPO（如 Anthropic 私募轮、OpenAI 营利转换）仍在司法争议中（Musk 庭审 Day 4 见速报）；Roze 反而走"机器人 + REIT + 芯片"实物资产组合 IPO 路径——**对汉坤代表中国 AI 基础设施客户（含寒武纪、中科曙光、紫光等）的赴美/赴港 IPO 打包结构有直接借鉴**——可整理一份"SoftBank Roze 资产打包模板研究"作为 PE/IPO 业务组内部分享；② **$5.4B + $6.5B + $3B = $14.9B 历史成本 → IPO 估值目标 $100B**——估值乘数约 6.7x，**远高于 ABB 机器人此前公开市场倍数（~2x）**；这一倍数差源自"AI 数据中心建造刚需"溢价，**汉坤代表中概股做反向收购或借壳上市的客户应注意"AI 概念注入"是否构成 SEC 审核中的可持续性问题**；③ **机器人替代建筑工人的劳工合规风险**——美国数据中心建造涉及大量本地劳工合同（含工会），Roze 大规模部署机器人施工**将引发美国数据中心所在州（VA、TX、AZ）劳工监管和工会诉讼**——汉坤代表中国 AI 公司在美建数据中心客户（如 ByteDance OCI、阿里云海外）应**预案"机器人替代工人引发的本地劳工反弹"**；④ **DigitalBridge REIT 注入意义**——美国数据中心 REIT 适用 IRC §856 特殊税制，**Roze IPO 后能否享受 REIT 税收优势是估值核心**——若 SEC/IRS 认定 Roze 为"集成 AI 基础设施运营商"而非纯 REIT，则估值需大幅折让；⑤ **孙正义本人风险标签**——SoftBank 历史上 IPO 估值高开低走（WeWork、Arm 等），**plaintiff bar 已对 Vision Fund 投后管理形成多起诉讼判例**——汉坤代表参与 Roze pre-IPO 轮的中国主权基金或 PE 客户应**做完整的"SoftBank 历史 IPO 投后回报"尽调备忘录**。

## ⑤ 本期技巧：用 mattpocock/skills 给律所的"半成型 AI 工作流"加上工程级护栏——尤其是 `/grill-with-docs` 与 `/tdd`

**场景：** 你是汉坤一位带 5-8 名年轻律师的合伙人，过去两个月所内已经把 Claude Code 用于合同条款修改、法律备忘录草拟、citation 核查。问题来了——年轻律师**给 AI 的 prompt 普遍写得太开放**："帮我改一下这条 indemnification 条款"——AI 自然会往大改方向走，可能把客户原本认可的措辞也"顺手优化"掉。同时你审完一份 AI 改的备忘录后**很难知道自己具体审了什么**——AI 没有事先列验收标准，你和它之间没有"已验证 vs 未验证"的明确分界。本周登顶 GitHub Trending 的 mattpocock/skills（**周新增 30,945 star，超越上期 Karpathy-skills 的 23K**）正好提供了两条针对性 skill——`/grill-with-docs`（开发前对齐）和 `/tdd`（先列验收标准再循环验证）——**这两条思路一字不改地搬到法律工作场景，能立刻消除上述两类痛点**。

**为什么好用：** Karpathy-skills 解决的是"AI 行为契约的总原则"（Simplicity / Surgical Changes / Clarity / Goal-Driven）；mattpocock/skills 走得更远——**把这些原则拆解为可调用的 slash command**，让 AI 在每次任务开始时**强制走"先对齐、后验证"的流程**。`/grill-with-docs` 的核心动作是：AI 不会立刻动手，而是反复用具体问题（参考相关 docs）"烤"提问者，直到对齐颗粒度足够细才进入执行；`/tdd` 的核心是：先把任务拆为"成功标准 1、2、3"清单，每完成一条立刻自检通过/不通过、不通过则回到上一步。**两条结合起来正好对应律师工作的两个核心痛点**——"客户和律师的需求颗粒度不一致"（前者）+"律师审核 AI 产出时缺乏验收标准"（后者）——而且**它的安装就是一行 `npx` 命令、与所内已有 CLAUDE.md（Karpathy 框架）完全兼容、可独立 commit 到 git 共享**，技术成本几乎为零。

**操作路径（4 步，30 分钟内完成）：**

1. **安装 mattpocock/skills 到所内主力项目目录：** 在 `~/projects/汉坤合同主力库` 或类似团队共享目录执行 `npx skills@latest add mattpocock/skills` ——会自动在 `.claude/skills/` 下生成全套 skill 文件并注册到 Claude Code；与上期 ⑤ 已部署的 Karpathy CLAUDE.md 不冲突
2. **修改条款类任务统一走 `/grill-with-docs` 启动：** 律师在 Claude Code 中开始任何"修改/改写"类任务前，**先输入 `/grill-with-docs <任务一句话描述>`**——AI 会反向追问 5-10 个细节（如"客户原文哪些措辞不能动？""赔偿上限是否有商业谈判余地？""管辖法适用何处？"），律师必须把答案给齐才进入执行——**这一步等于把'客户认可的原文'白纸黑字钉在上下文里，AI 后续不会越界改动**
3. **审核类任务统一走 `/tdd` 验收：** 当 AI 完成一份草稿后，律师**输入 `/tdd 列出本备忘录应满足的 5 条验收标准并逐条自检`**——AI 会先列清单（如"引用判例必须从 Westlaw 粘贴""赔偿条款必须保留客户原措辞""时区表述不得擅自标准化"），然后**逐条标注通过/未通过**，未通过的部分要求律师手动修复或确认放行——**这一步等于把过去律师"凭感觉审"升级为"按清单审"**
4. **配合 `/diagnose` 处理"AI 出错怎么办"：** 当律师发现 AI 产出有具体问题（如引文格式错误、条款编号跳号），输入 `/diagnose <具体问题>`——AI 会按结构化方法找根因（vs 直接打补丁式硬改），找到根因后再让 AI 给出"最小修改方案"——**对法律文书的可追溯性极重要**
